# 微调学习笔记

## 1. 今天微调的目标

本次微调不是训练企业知识，而是训练模型的行为模式。

当前目标是：

```text
用户自然语言
  -> 模型输出稳定 JSON
  -> Agent 根据 JSON 决定走 RAG / Tool / Chat / Approval
```

例如：

```text
用户：帮我查一下 INC-1001 状态
模型：{"intent":"ticket_query","ticket_id":"INC-1001","priority":null,"category":"ticket"}
```

这更接近 Agent 里的 intent classifier。

企业知识不要直接微调进模型，应该继续走 RAG。

一句话：

```text
知识用 RAG，行为模式用微调。
```

## 2. train 和 valid 在做什么

`train` 是训练集。

模型会反复看这些样本，学习规律：

```text
“查 INC-1001 状态”
  -> ticket_query

“创建高优先级 VPN 工单”
  -> ticket_create

“邮箱登录失败怎么办”
  -> knowledge_question

“查 LAP-7788 资产”
  -> asset_query
```

训练时，模型会根据 user 内容预测 assistant 内容，然后和标准答案比较。

如果预测错了，就通过反向传播更新 LoRA adapter 参数。

简单说：

```text
train = 给模型做练习题，并根据答案改参数
```

`valid` 是验证集。

验证集不参与参数更新。它的作用是检查模型有没有真的学会，而不是只背训练题。

简单说：

```text
valid = 考试题，只看成绩，不改参数
```

## 3. messages 里的 assistant.content 是什么

训练数据里通常长这样：

```json
{
  "messages": [
    {
      "role": "system",
      "content": "You are an enterprise IT intent extraction assistant. Return compact JSON only."
    },
    {
      "role": "user",
      "content": "帮我新建一个中优先级网络工单，会议室网络断开"
    },
    {
      "role": "assistant",
      "content": "{\"intent\":\"ticket_create\",\"ticket_id\":null,\"priority\":\"medium\",\"category\":\"network\"}"
    }
  ]
}
```

其中：

```text
system    = 规则
user      = 题目
assistant = 标准答案
```

`assistant.content` 里面就是模型要模仿输出的标准答案。

实际目标文本是：

```json
{"intent":"ticket_create","ticket_id":null,"priority":"medium","category":"network"}
```

含义：

```text
intent: ticket_create
表示这是创建工单意图

ticket_id: null
因为是新建工单，还没有现成的工单号

priority: medium
因为用户说“中优先级”

category: network
因为用户说“网络工单 / 会议室网络断开”
```

## 4. LLM SFT 和 CNN 训练的类比

你有 CNN / GAN / 图像处理背景，可以这样类比：

```text
CNN 图像分类：
输入图片 -> label
模型预测 -> 交叉熵 loss -> 反向传播

LLM SFT：
输入 prompt tokens -> assistant answer tokens
模型预测下一个 token -> token-level cross entropy loss -> 反向传播
```

区别是：

```text
CNN 的 label 通常是类别 ID
LLM 的 label 是一串 token
```

比如这条答案：

```json
{"intent":"ticket_create","ticket_id":null,"priority":"medium","category":"network"}
```

训练时不是整体一次判断对错，而是逐 token 预测。

每个 token 都会参与 loss。

## 5. LoRA 是不是大模型预训练时预留的

不是。

LoRA 不是基础模型原生留好的“插槽”，而是微调时人为改造模型结构。

它会把原来某些线性层临时包装一下，在旁边加两个很小的低秩矩阵。

以一个线性层为例，原来是：

```text
y = W x
```

LoRA 之后变成：

```text
y = W x + ΔW x
```

其中：

```text
W    = 原始大模型参数，冻结，不训练
ΔW   = LoRA 额外学出来的小参数
```

但 LoRA 不直接训练完整的 `ΔW`，而是把它分解成两个小矩阵：

```text
ΔW = B A
```

所以变成：

```text
y = W x + B A x
```

其中：

```text
A = 降维矩阵
B = 升维矩阵
```

假设原来 `W` 是：

```text
4096 x 4096
```

全量训练需要更新：

```text
4096 * 4096 = 16,777,216 个参数
```

如果 LoRA rank `r=8`：

```text
A: 8 x 4096
B: 4096 x 8
```

需要训练：

```text
8 * 4096 + 4096 * 8 = 65,536 个参数
```

参数量少很多。

所以它不是预训练模型提前留好的，而是 PEFT / LoRA 框架在加载模型后，把指定层替换或包装成支持 LoRA 的层。

例如训练脚本里写了：

```python
target_modules=[
    "q_proj",
    "k_proj",
    "v_proj",
    "o_proj",
    "gate_proj",
    "up_proj",
    "down_proj",
]
```

意思是：在这些 Transformer 线性投影层旁边加 LoRA 分支。

可以理解为：

```text
原模型：
Linear(q_proj)

LoRA 后：
Linear(q_proj, frozen)
+ LoRA_A
+ LoRA_B
```

训练时：

```text
原来的 q_proj 权重不动
只更新 LoRA_A / LoRA_B
```

推理时：

```text
基础模型 + LoRA adapter 一起生效
```

一句话：

```text
LoRA 不是模型预留接口，而是微调框架动态给模型的某些线性层外挂了一组可训练低秩参数。
```

## 6. 训练结束后为什么要 eval

训练结束不代表模型已经可用。

训练结束只说明：

```text
数据 -> SFT -> LoRA adapter -> 保存产物
```

这条链路跑通了。

接下来要做 eval，也就是拿模型没见过或少见的样本测试它。


它会测试几类问题：


并检查模型输出是否符合我们定义的 JSON schema。

```text

```

这不是训练链路失败，而是说明模型还没有学稳我们的输出协议。

观察到的错误模式：

```text
1. 模型会输出 JSON，但字段名不稳定
2. 模型会自己发明 intent 值
3. 模型会自己发明字段，例如 intent_type、query_type、asset_id
4. ticket_create 场景里，模型甚至会编造 ticket_id
5. Qwen3 会输出 <think> 标签，需要后续处理
```

这说明第一版 adapter 目前只学到了：

```text
回答应该像 JSON
```

但还没有学稳：

```text
必须严格输出我们定义的 JSON schema
```

下一轮优化方向：

```text
1. 增加训练数据量
2. 给每个 intent 增加更多中文和英文表达
3. 加入更多负例和边界样本
4. 在 system prompt 里明确允许的 intent 枚举和值
5. 在 eval 里固定检查字段和值
6. 必要时训练更久或换更适合 instruction following 的模型
```

这就是微调最真实的工作流：

```text
train
  -> infer
  -> eval
  -> analyze failures
  -> add data / adjust prompt / tune params
  -> train again
```

## 8. 微调模型怎么接回 Agent

微调模型不能直接裸接 Agent。

原因是模型输出仍然可能出现：

```text
<think> 标签
非法 JSON
额外字段
错误 intent
编造 ticket_id
大小写不统一
```

所以中间必须有一层 contract，也就是输出合同层。

本次新增：

```text
E:\agent\fine_tuning\scripts\intent_contract.py
E:\agent\fine_tuning\scripts\agent_intent_preview.py
```

`intent_contract.py` 负责：

```text
1. 从模型原始输出中提取 JSON
2. 只保留允许的字段
3. 校验 intent / priority
4. 规范化 ticket_id / asset_tag
5. ticket_create 时强制清空 ticket_id，防止模型编造
6. 返回 validation_failures，便于 fallback 或审计
```

`agent_intent_preview.py` 用来预览微调模型输出如何进入 Agent。

运行示例：

```powershell
cd E:\agent\fine_tuning
& E:\agent\enterprise-it-agent\.venv\Scripts\python.exe .\scripts\agent_intent_preview.py "Create a medium priority network ticket. Meeting room wifi is down."
```

输出结构：

```json
{
  "user_query": "...",
  "raw_model_output": "...",
  "extracted_json": {},
  "normalized_for_agent": {},
  "validation_failures": [],
  "agent_route": "ticket_create"
}
```

接入 Agent 的推荐流程：

```text
用户问题
  -> fine-tuned intent model
  -> extract_json
  -> normalize_intent_payload
  -> validate_intent_payload
  -> 如果通过：LangGraph 按 intent 路由
  -> 如果失败：fallback 到原来的 LLM intent classifier 或 smalltalk/knowledge_question
```

也就是说：

```text
fine-tuned model 负责建议
contract layer 负责把关
LangGraph 负责执行
```

不要让模型输出直接决定高风险动作。

尤其是：

```text
ticket_create
approval
asset lookup
permission-sensitive RAG
```

这些场景必须继续保留校验、审批、审计。

## 9. 微调模型接入 Agent 的推荐架构

微调模型不是系统核心，协议才是核心。

推荐结构：

```text
User Query
  -> Intent Provider
      -> Fine-tuned local model
      -> 或原来的 LLM classifier
      -> 或规则兜底
  -> Contract Layer
      -> JSON extraction
      -> schema validation
      -> normalization
      -> risk checks
  -> LangGraph State
  -> Route
      -> RAG
      -> Tool
      -> Approval
      -> Chat
```

核心思想：

```text
模型可以换
协议不能乱
```

以后可能会换：

```text
Qwen3-0.6B -> Qwen3-1.7B
LoRA v2 -> LoRA v3
本地模型 -> API 模型
SFT -> prompt-only classifier
规则分类 -> hybrid classifier
```

但只要它们最终都输出同一个 contract：

```json
{
  "intent": "ticket_query",
  "ticket_id": "INC-1001",
  "asset_tag": null,
  "priority": null,
  "category": "ticket"
}
```

后面的 LangGraph 就不用大改。

三层理解：

```text
模型输出层：
  原始输出，可能不稳定

Contract Layer：
  把不稳定输出变成稳定 DTO

Agent State：
  LangGraph 用稳定字段做路由和执行
```

不要让系统直接依赖 raw model output。

不要这样：

```python
if "ticket" in raw_model_output:
    run_ticket_tool()
```

应该这样：

```python
payload = extract_json(raw_model_output)
normalized = normalize_intent_payload(payload)
failures = validate_intent_payload(normalized)

if failures:
    fallback_to_llm_classifier()
else:
    route_by_intent(normalized["intent"])
```

企业级 Agent 的原则：

```text
模型可替换
协议稳定
校验前置
执行后置
审计保底
```

## 10. Intent Provider 抽象

`Intent Provider` 是意图识别能力的接口层。

它的目标是：

```text
让 LangGraph 不直接依赖某一个具体模型
```

可以类比 Java：

```java
public interface IntentProvider {
    IntentContract classify(String question);
}
```

Python 里可以用 `Protocol` 表达：

```python
class IntentProvider(Protocol):
    def classify(self, question: str) -> IntentContract:
        ...
```

不同 provider 都实现同一个方法：

```text
FineTunedIntentProvider
  用本地 Qwen + LoRA

PromptIntentProvider
  用原来的 LLM classifier

RuleBasedIntentProvider
  用正则/关键词兜底

FallbackIntentProvider
  组合多个 provider
```

推荐链路：

```text
用户问题
  -> FineTunedIntentProvider
      -> 成功：返回 IntentContract
      -> 失败：进入下一层
  -> PromptIntentProvider
      -> 成功：返回 IntentContract
      -> 失败：进入下一层
  -> RuleBasedIntentProvider
      -> 成功：返回 IntentContract
      -> 失败：默认 knowledge_question
```

Provider 输出不要是原始模型文本，而应该是稳定 DTO：

```python
class IntentContract(BaseModel):
    intent: Literal[
        "knowledge_question",
        "ticket_query",
        "ticket_create",
        "asset_query",
        "smalltalk",
    ]
    ticket_id: str | None = None
    asset_tag: str | None = None
    priority: Literal["low", "medium", "high"] | None = None
    category: str | None = None
    provider: str
    raw_output: str | None = None
    validation_failures: list[str] = []
```

这些字段的作用：

```text
intent
  给 LangGraph route 用

ticket_id / asset_tag / priority / category
  给 tool node 或 RAG filter 用

provider
  审计和排查用

raw_output
  debug 用

validation_failures
  判断是否 fallback
```

示例代码：

```text
E:\agent\fine_tuning\scripts\intent_provider_demo.py
```

运行：

```powershell
cd E:\agent\fine_tuning
& E:\agent\enterprise-it-agent\.venv\Scripts\python.exe .\scripts\intent_provider_demo.py
```

一句话：

```text
Intent Provider 是模型能力适配层；
Contract Layer 是输出安全边界；
LangGraph 是业务流程编排层。
```

```text
3. 在 LangGraph classify node 中增加 provider 配置开关
4. 保留 fallback 到原 classifier
5. 对 ticket_create / asset_query 保留审批和审计
```

## 15. A/B 对比：原 LLM classifier vs 微调 intent

新增 A/B 对比脚本：

```text
E:\agent\fine_tuning\scripts\ab_compare_intent.py
```

它会使用同一批 eval case：

```text
E:\agent\fine_tuning\data\intent_eval_cases.jsonl
```

分别调用：

```text
POST /intent
POST /fine-tuned-intent/preview
```

运行：

```powershell
cd E:\agent\fine_tuning
& E:\agent\enterprise-it-agent\.venv\Scripts\python.exe .\scripts\ab_compare_intent.py
```

当前 A/B 结果：

```text
total: 40

原 LLM classifier:
  intent accuracy: 40/40 = 100%

微调 LoRA intent:
  intent accuracy: 38/40 = 95%
  full field accuracy: 28/40 = 70%
```

解释：

```text
原 LLM classifier 只评估 intent，因为它原本只返回 intent/reason。
微调模型评估两层：
  intent accuracy：只看 intent 对不对
  full field accuracy：intent、ticket_id、asset_tag、priority、category 全部字段都要对
```

结论：

```text
原 LLM classifier 泛化更强
微调模型已经能稳定输出结构化字段，但字段级准确率还需要继续提升
```

微调模型主要失败点：

```text
1. "has been resolved" 被误判为 knowledge_question
2. 中文“开一个邮箱工单”被误判为 ticket_query
3. category 边界不稳定：wifi/network、outlook/email、account/email、device/asset
4. smalltalk 的 intent 基本对，但 category 常输出 general/thankyou，而不是 chat
```

工程建议：

```text
短期：
  不要直接替换原 LLM classifier
  用微调模型做候选 Intent Provider
  失败或低置信时 fallback 到原 LLM classifier

中期：
  扩充 v3 训练数据
  对 category 做归一化映射
  再跑 A/B

长期：
  做 provider 配置化：
    INTENT_PROVIDER=llm
    INTENT_PROVIDER=fine_tuned
    INTENT_PROVIDER=fallback
```

## 16. LangGraph 使用可配置 Intent Provider

已在 `enterprise-it-agent` 中实现：

```text
INTENT_PROVIDER=llm
INTENT_PROVIDER=fine_tuned
INTENT_PROVIDER=fallback
```

默认值：

```text
llm
```

配置位置：

```text
E:\agent\enterprise-it-agent\app\core\config.py
```

新增 provider 层：

```text
E:\agent\enterprise-it-agent\app\fine_tuning\intent_provider.py
```

LangGraph 的 `classify_intent_node` 现在不直接调用原来的 `classify_intent`，而是调用：

```python
classify_with_configured_provider(question)
```

三种模式：

```text
llm
  使用原来的 LLM classifier

fine_tuned
  使用本地 Qwen3-0.6B + LoRA v2

fallback
  先使用 fine_tuned
  如果 contract/guardrail 失败，再 fallback 到 LLM classifier
```

验证方式：

```powershell
cd E:\agent\enterprise-it-agent

$env:INTENT_PROVIDER='fine_tuned'
& E:\agent\enterprise-it-agent\.venv\Scripts\python.exe -m uvicorn app.main:app
```

然后调用：

```text
POST /graph/run
```

请求：

```json
{
  "user_query": "Check INC-9012 status."
}
```

期望 state 中出现：

```json
{
  "intent": "ticket_query",
  "intent_provider": "fine_tuned",
  "intent_contract": {
    "ticket_id": "INC-9012",
    "category": "ticket",
    "validation_failures": []
  },
  "route_name": "tool",
  "tool_name": "get_ticket_status"
}
```

本次还加入了轻量 semantic guardrails：

```text
问题包含 INC-xxxx，但模型没有输出 ticket_query -> 触发 fallback
问题包含 LAP/ASSET 标签，但模型没有输出 asset_query -> 触发 fallback
问题明显是创建工单，但模型没有输出 ticket_create -> 触发 fallback
```

示例：

```text
Can you tell me whether INC-1200 has been resolved?
```

微调模型曾误判为：

```text
knowledge_question
```

现在 `fallback` 模式会识别出问题中包含 `INC-1200`，触发回退到 LLM，最终得到：

```text
ticket_query
```

核心结论：

```text
fine_tuned 适合低成本、结构化、稳定场景
llm 适合泛化和复杂语义
fallback 是当前最稳妥的企业级接入方式
```
